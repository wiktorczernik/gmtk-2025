Getting started
The Multiversal Vehicle Controller is a Unity toolkit made for ambitious game developers to integrate vehicle physics inside their video-games. You can download the Multiversal Vehicle Controller at mvc.gg .
The MVC User Manual helps you know how to use our toolkit and its features. Read it from start to finish, or use it as a reference. In case you're looking for a way to implement the MVC scripts with your own, make sure you check our API Scripting Reference for more information.
If it's your first time using the Multiversal Vehicle Controller, you can check our YouTube channel , where you can find some helpful tutorials to make your first steps.
How to install
Installing the Multiversal Vehicle Controller is an easy process that takes a few minutes.
After downloading the toolkit from the Unity Asset Store or mvc.gg . All you need to do next is import the downloaded Unity package into your Unity project which is named as mvc_pro_1.4.1-beta.x.unitypackage or mvc_community_1.4.1-beta.x.unitypackage
To import a Unity package, go to the following menu Assets/Import Package/Custom Package..., then select the downloaded package and click the Open button and wait for the Unity Editor to decompress the requested package.
A window will popup, the next thing you need to do is make sure that everything is checked, then click the Import button of the Unity Package Importer.
Once everything has been compiled the Install Wizard window will appear, all you need to do is accept the requested data and wait for it to import its requirements into your project.
If you were prompted to restart the Unity Editor, click on Restart, Apply or Yes. If the Install Wizard do not appear when the Unity Editor is restarted, all you've to do it go to Tools > Install MVC and you'll be good to go!
After fetching & installing the required packages, all is left to do is click Start Install.
Setup wizard
The Setup Wizard is a cool feature made for the MVC to make understanding the toolkit an easy process for our users.
Once the toolkit has been installed successfully, the Setup Wizard window will popup. Next thing to do is click the Let's get started button.
The next to appear is the Configurations panel. There're a couple of things to choose depending on your needs mentioned below...
  • Render Pipeline: The render pipeline used in your project, somehow it's detected automatically and can be changed, but after making some modifications to your project settings.
    The available render pipelines are the following: Standard (Built-in RP ), Universal Render Pipeline, High Definition Render Pipeline and Custom Render Pipeline.
  • Physics: The vehicle wheel physics, by default it's Wheel Collider, other physics like Volumetric Wheel will be available in later versions of the Multiversal Vehicle Controller. Read more...
  • Agent Type: The agent type that will be used by the toolkit's AI systems.
    • Using the built-in Humanoid agent settings is not recommended. You'll have to create your own agent type by using the button. You can also click on Show Recommended Agent Settings to see the best recommended you can use when creating a new agent type.
  • Vehicles Layer: The layer that MVC uses to regroup its GameObjects.
  • Wheels Layer: A layer used to regroup any available wheels GameObjects.
    • Vehicles Layer and Wheels Layer must be assigned to a layer different than the built-in Default layer.
    Vehicles Layer and Wheels Layer values cannot match.
  • Walkable Mask: A layer mask for AI walkable objects.
  • Non-Walkable Mask: A layer mask for AI non-walkable areas, objects and obstacles.
    • Walkable Mask and Non-Walkable Mask cannot have common layers, keep in mind that the Vehicles Layer and Wheels Layer cannot added to the mentioned masks as this can cause runtime issues.
    Non-Walkable Mask can be empty, unlike Walkable Mask which has to have at least one layer added including the built-in Default layer.
    You can click on Add Recommended Layers to add and setup layers for your project automatically.
  • Runtime Units: To know more about this field, read this section...
  • Editor Units: Just as the Runtime units, the Editor Units are the same, except that the expressed values appear in the MVC component inspectors and within the Settings panel.
  • Inputs System: The Inputs System is a system that manages the user's inputs like a keyboard keys or a joystick's. The MVC uses two options:
    • Inputs Manager: An input system made our team for improved and easy input setup based on the New Unity Input System to manage keyboard and joystick inputs at once.
    • Unity Legacy Input System: Is the original Unity input system, it's stable but missing a lot of features that the Inputs Manager provides.
  • Audio Folder Path: Is the path the MVC should save audio files and sound effects, the folder should be inside a Resources folder, so the MVC can load them at runtime.
    • Click on Use Suggested Path to use the recommended path suggested by the toolkit.
After assigning all of the needed fields, click Next. A confirmation message will appear, click Yes to proceed with the setup process.
You might be prompted to import the Getting Started Demo Packages before starting the setup process.
Once the import process is done, a success message should appear. To close the panel, click Open Settings to show the Settings panel.
Render pipeline
A render pipeline performs a series of operations that take the contents of a Scene, and displays them on a screen. To learn more about render pipelines here's a Unity documentation section that you can make use of.
Once you choose a different option than the default one from the Render Pipeline list. An error message will appear:
This project doesn't have a <Your New Render Pipeline Option> Asset added to its graphics settings! Please consider changing the render pipeline option above or go to `Edit > Project Settings > Graphics` and add a Scriptable Render Pipeline Asset.
To fix this issue, you need to go to Edit > Project Settings > Graphics and assign an SRP Asset . Here are some Unity documentation articles about how to create a new SRP Asset for each render pipeline separately.
  • URP: Universal Render Pipeline
  • HDRP: High-Definition Render Pipeline
  • C-RP: Custom Render Pipeline
In case you chose the Standard option, the error message will differ from the one above! The Built-in Render Pipeline doesn't require an SRP Asset , in fact you can just empty the Asset field within the Graphics Settings panel and the error message will disappear. In case you got some visual issues because of the loaded shaders, you can just remove the installed Render Pipeline package from your project and everything will look good again.
Audio folder path
One of the best features the MVC provides, is the incredible SFX System. For that, a folder is needed to save and load the required sound effects and audio clips for the toolkit.
The Audio Folder Path is the last field to fill-in. To do that, you need to click on the ... button, and a popup window will appear for you to choose a folder. After creating your own folder, select it and click Select Folder.
The new audio folder has to be inside the Resources folder of your project. Example: <Project Folder>/Assets/Resources/.

Settings panel
The Settings panel is where most of the magic happens, it organises all of the MVC systems' configurations.
To open it you need to go to Tools > Multiversal Vehicle Controller > Edit Settings..., and the Settings panel window will appear.
On the left side of the panel you can find the different settings tabs, that's where you can configure each system on it's own. Check the following sections for more information.
General
General settings are dedicated for the most basic configurations of the Multiversal Vehicle Controller, where you can change the physics system, steering method, layers, tags...
Check the rest of this section below to learn more about each configuration separately.
Physics System
The Physics System/Model is the one responsible for moving your vehicles inside Unity, and each system has its own behaviour style.
  • Wheel Collider: Basing on the Unity standard Wheel Colliders , the MVC uses this component to simulate vehicles movements and expands the number of available values for an advanced and a precise behaviour. The Wheel Colliders are based on the base physics of Unity, Nvidia PhysX .
  • Volumetric Collider: The Volumetric Wheel system is a much more sophisticated version of a simple Raycasts system. With the use of C# Jobs package of Unity, this feature has optimal performance.
Sub Steps
During each Fixed Update in the vehicle simulation, the fixed delta time is divided into smaller sub-steps. These sub-steps allow for the calculation of suspension and tire forces for each smaller delta. The resulting forces and torques are then summed up, integrated, and applied to the vehicle's Rigidbody .
The stability of the simulation and, consequently, the stability and handling of the vehicle are controlled by these sub-steps. Adding more sub-steps improves the stability of the vehicles but also introduces a higher performance impact at runtime.
Steering Method
Steering is one of the most important functions of a vehicle, for that our toolkit provides different steering methods which are the following:
  • Simple: A basic straight-forward steering behaviour basing just on the user inputs.
  • Dynamic: A complex steering behaviour dedicated for a better anti-slip behaviour on higher speeds.
  • Ackermann: Is a geometry method commonly used in the Automotive industry to improve handling at lower and higher speeds. Read more...
Transmission
One of the most used features used in video-games is transmission/gearbox workflow. The Multiversal Vehicle Controller provides the following:
  • Automatic: As the name suggests, Automatic is an automated behaviour of a certain system, in this case and while the Unity Editor is in Play mode, the user will just have to move the vehicle freely and the its transmission will shift its gears without a single input.
  • Manual: This type of transmission behaviour is the total opposite of the Automatic one, the player will have shift gears on his own while moving the vehicle.
Wheel Renderer Refresh
This controls how the wheels rotation angles should be calculated.
  • Accurate: Calculates the wheels rotation angles precisely.
  • Approximate: Gives the wheels rotation angles approximate values.
Runtime Units
While the Unity is in play mode, some values may appear on the screen UI. Example: Speed, torque, power & weight... These values can be expressed in Metric units like KM/H , N.m , hp , Kg , m and L . Or in Imperial with units like MPH , lb⋅ft , Kw , lbs , ft and gal .
Power Unit
The runtime unit used for Power values.
Torque Unit
The runtime unit used for Torque values.
Auto Engine Shutdown
This feature will automatically shutdown the player's vehicle engine when stationary.
Engine Shutdown Timeout
The amount of time is take the vehicle engine to be shutdown.
Start Engines at Awake
Once the Unity Editor enters play mode, all of the vehicles engines will start up and they'll be controllable, once the Unity Editor enters play mode. Note that this feature is not available for AI vehicles.
In case Start Engines At Awake is disabled, the player will be required to turn on his vehicle engine manually using the Turn On/Off Engine input.
You can change the Turn On/Off Engine input using the Change Input button. Also, you can modify the current input inside the Inputs Manager by clicking on Edit Input.
Fuel/Battery System
For a much realistic behaviour, you can add fuel/battery consumption while a vehicle is running.
Engine Stall
For a realistic drivetrain and clutch behaviour of your vehicles, you can turn on this feature.
Here's how it works: Let's suppose you're in a car, you turn it on, put your foot on the clutch pedal, change from neutral to 1st gear and release your foot off the clutch immediately... Obviously the first is going to happen is that your engine is going to shutdown. This feature allows for such behaviour to happen.
This feature is not available for EVs or vehicles with automatic gearboxes.
Gear Limiter (First Gear/Reverse)
Gear limiter is a feature that prevents an Engine to Over-Rev or Under-Rev by adding brake/motor torque to the wheels, disabling this features (for the first/reverse gear only) will lead for the car stop automatically when not pressing the fuel pedal, but in any case, the engine RPM won't pass the engine's Redline RPM.
Burnouts For Auto Transmission
For vehicles with Automatic Gearboxes, enabling this allows the player to do Tire Burnouts . In reality most car manufacturers that make automatic gearboxes don't allow drivers to do such stunts, burnouts are kind of exclusive to vehicles with Manual Gearboxes, since drivers have control over the Clutch through the Clutch Pedal.
Vehicles Interior
Interior components like steering wheels, speedometer/tachometer help improve the gaming experience. Turning this feature on will allow you to edit these kinds of components from your Vehicle inspector.
Reset Vehicles
In case your vehicle flipped over or crashed into something, the reset feature respawns the player's vehicle or any other AI vehicles into their stationary state. You can configure the reset system through the following fields:
Reset Timeout
How much time it takes for the reset system to detect any flipped vehicles to respawn them.
Reset Cooldown Time
The time it takes the reset system to re-enable collision detection with other objects or vehicles after respawning.

Artificial Intelligence (AI)
The following section and all of its parameters are dedicated to vehicles with AI controllers.
AI Layers
Learn more about this feature in the following section.
Agent Type
The preferred Unity AI Agent for baking AI Surfaces within the Vehicle Manager inspector. You can edit the currently used agent by clicking the button on the right side of the mentioned parameter. You can also show the recommended settings if you're willing to create a new agent, as using the default Humanoid agent settings is not recommended.
Smooth Steering
Smooth Steering gives the AI behaviour a bit of a realistic flavour, instead of instantly applying the steering calculations of the AI Controller, this feature improves the output to be better, visually and in the back-end.
Steer Intensity
Refers to the steering speed intensity. Higher values cause a more responsive behaviour of the wheels.
Vehicle Path Distance
The required distance between an AI vehicle and a path point for the AI Controller to switch his focus on the next point of the AI Path.
The Vehicle Path Distance has two modes:
  • Dynamic: In this mode, the Distance value is an interval that has a Minimum and a Maximum for certain speeds.
    • Min: The vehicle to path point distance at 0 Kilometres Per Hour (km/h) / 0 Miles Per Hour (mph).
    • Max: The vehicle to path point distance at 100 Kilometres Per Hour (km/h) / 62 Miles Per Hour (mph).
  • Fixed: Where the Distance will be a fixed value without any ranges unlike the Dynamic mode.
Path Point Spacing
The distance between each AI Path point.
Reverse Timeout
How much time it takes AI Vehicles to engage the Reverse gear when it's stuck and can't move forward to its target position anymore.
Reverse Time Span
The amount of time it takes AI Vehicles to disengage the Reverse gear.
Agent Refresh Timeout
A time span for AI agents to refresh their destination path.
Agent Refresh Timeout
A time span for AI agents to refresh their destination path.
Path Avoidance Curve
A smoothness curve for AI Path components used to avoid obstacles. The linear the curve is, the sharper AI Paths avoid obstacles.
The Path Avoidance Curve limits cannot exceed 0 and 1 on both axes, Time (X) and Value (Y), otherwise the curve will be stretched or clamped automatically at Runtime which is not recommended.
Obstacle Detection
The following parameters of this sub-section, have control over AI obstacle detection & avoidance.
You can set the detection & avoidance quality of vehicle obstacles by using the enumerator on the right side of this parameter. The better the quality, the low the performance becomes.
Some AI features might be disabled for some range of qualities.
Geometry Type
The type of geometry system used to treat static GameObjects as obstacles.
Sensors/Front Sensors
To avoid obstacles, AI Vehicles need sensors to detect them.
Front Sensors are the main sensors of any AI Vehicle and have two modes:
  • Dynamic: This mode allows the toolkit to change the sensors length & angle depending on their vehicle speed.
  • Fixed: By enabling this mode, the sensors length will be static and won't change.
Sensors Steer
This feature allows the Front Sensors to steer in relation to their AI Vehicle steering angle.
Sensors Steer Multiplier
The sensors steering angle multiplier.
Sensors Length
The front sensors length.
The type of sensors length depends on the Sensors Mode. Here's a small recap for each mode:
  • Dynamic: An interval value with a minimum and a maximum.
    • Min: The minimum sensor length at 0 Kilometres Per Hour (km/h) / 0 Miles Per Hour (mph).
    • Max: The maximum sensor length at 100 Kilometres Per Hour (km/h) / 62 Miles Per Hour (mph).
  • Fixed: A single static number value.
Clamp Sensors Length
Setting this parameter to No allows the sensors length to exceed to its Maximum value.
This parameter is only available for Dynamic sensors.
Sensors Angle
The front side sensors angle.
The type of sensors angle depends on the Sensors Mode. Here's a small recap for each mode:
  • Dynamic: An interval value with a minimum and a maximum.
    • Min: The minimum sensor angle at 100 Kilometres Per Hour (km/h) / 62 Miles Per Hour (mph).
    • Max: The maximum sensor angle at 0 Kilometres Per Hour (km/h) / 0 Miles Per Hour (mph).
  • Fixed: A single static number value.
The sensors angle is always clamped between 0 and 90 degrees.
Clamp Sensors Angle
Setting this parameter to No allows the sensors angle to exceed to its Maximum value.
This parameter is only available for Dynamic sensors.
Setting the Maximum Sensor Angle to 90 degrees and enabling the Sensors Angle Clamp toggle won't make any change to the sensors behaviour. This only applies to Dynamic sensors.
Side Sensors Length
The length of side sensors for AI Vehicles.

Behaviour & Physics
The Multiversal Vehicle Controller is all about physics, when it comes to that, we try to deliver the best advanced sets of configuration so developers can make the most of our systems and tweak values as they want. For this, the MVC provides a list of fields as mentioned below in this section.
Aerodynamic Wings
Enabling this toggle allows you to use vehicle's Aerodynamic Wings.
Downforce
When you enable Downforce, you get to use it within the Stability settings and when you configure your vehicle's Active Aerodynamics.
Air Anti-Roll
In case this feature is enabled: When a vehicle is flying in the air and it's on its way to flip, the Air Anti-Roll adds some forces to try to fix this issue and get rid of un-wanted flipping behaviours. This feature is not very suitable for simulation physics, since it adds forces to neglect realistic behaviours caused by the physics systems.
Air Anti-Roll Intensity
The intensity/multiplier of forces applied to the vehicle.
Non-Ground Downforce
Should the vehicle's Stability system and its Aerodynamics add downforce when it's not collided with a ground object. This feature is suitable for simulation physics.
Drag
Drag usually means resistance, and in this case we're talking about air resistance at higher speeds.
Wheel Mass
Neglecting wheels' mass from the overall vehicle mass has an effect on the vehicle's centre of mass and it's behaviour in general, this causes a realistic behaviour because of the happening back-end calculations.
Engine Mass
This feature is the same as the Wheels Mass feature, except that it has a much more noticeable effect in the vehicle's behaviour and relies on the Engine Position configuration of the Chassis component and on the mass of the vehicle's engine.
Steer Intensity
The mentioned field controls the player's steering input's intensity, the higher the value, the intense the vehicle steering, the lower this value is, the slower the vehicle steering responsiveness.
Steer Release Intensity
Just like Steer Intensity, Release Intensity is meant to take control when the Steering Wheel input has been released or is inverting direction.
Steer Interpolation
The way in which the steering value should interpolate over time.
  • Instant: The steering value will instantly change to the target value.
  • Linear: The steering value will move constantly over time.
  • Exponential: This will let the steering value reach its target in a Logarithmic Form (i.e. fast at first and slow when reaching the target value).
Maximum Steer Angle
The maximum steering angle value for every vehicle.
Minimum Steer Angle
The minimum steering angle value for every vehicle.
Suspension Adjustments
Enabling this feature allows you to customize the vehicle's suspension stancing values, such as Camber , Caster and Toe .
Maximum Camber
The suspension's maximum camber angle for every vehicle.
Maximum Caster
The suspension's maximum caster angle for every vehicle.
Maximum Toe
The suspension's maximum toe angle for every vehicle.
Maximum Side Offset
The rim maximum side offset for every vehicle.
Side Offset Affect Handling
Should Side Offset values affect a vehicle physics handling?
Launch Control
Launch Control is a feature commonly used in the automotive industry to get the best launch time out of a vehicle. Launch control usually acts like a normal RPM rev-limiter, but instead of limiting the RPM to the corresponding engine's Over-Rev RPM, this feature limits the vehicle's revving range to the engine's torque peak RPM when it's enabled. The mentioned feature is disabled by default at runtime, you can activate it using the Launch Control Input.
Counter Steering
Counter Steering is a stability feature that acts when a vehicle's rear wheels are slipping on one of their sideways axes. The counter steering system steers the vehicle automatically to the opposite of the vehicle's slipping direction to maintain stability.
Counter Steering Priority
This selection field is responsible for how the counter steering system should work, each option is described below:
  • Steering Wheel: The following option prioritizes the player's input over the counter steering system stability input.
  • Average: Using this option will let the steering system calculates the average between the counter steering system input and the player's input and applies the result value to the steering wheels.
  • Counter Steer: This option prioritizes the counter steering system's input over the player's steering input.
Counter Steer Intensity
The counter steer angle input multiplier.
Counter Steering Damping
This field smooths out the counter steering system angle input, the lower the value, the slower the counter steering to react and vice-versa.
NOS
Nitrous, commonly spelled as NOS thanks to the Fast & Furious culture, is a method to boost engines in order to get more power and torque by adding a denser air composition called Nitrogen-dioxide (NO₂) inside the engine's air intake in order to get an efficient fuel burning process. NOS is usually stored inside bottles, which they differ in size, number and boost efficiency.
This feature is not available for EVs , since they use electric motors instead of internal combustion engines.
NOS Reloading
Enabling this feature will allow the player to get reproduceable amounts of NOS after consumption.
NOS Delay
When both of the NOS Reloading and this feature are enabled, the reload system will have wait a for specific time span before regenerating NOS.
NOS Delay
How much time does it take to start reloading the NOS bottles after last use.
Progressive NOS
Progressive NOS has a huge effect on NOS bottles performance, the lower the NOS amount inside a specific NOS bottle, the lower the boost is and vice-versa.
One Shot NOS
When enabled, once the player activates the NOS Input, he'll not be able to cancel boosting his vehicle's engine unless the currently used NOS bottle becomes fully empty.
Damage
The Damage system in its beta phase is still pretty basic and simple, we're working on improving it in future releases of our toolkit! Until that time, here's a full list of the available fields and configurations.
Velocity Levels
The damage system used velocity levels to determine the intensity of the vehicle's collision or impact with other objects or vehicles. We can identify three levels: Low, Medium and High.
Wheel Health
Wheels Health is a feature that has been included in the damage system to simulate tires health, for example a tire can blow up because of wearing out due to doing skids and donuts . That can cause wheel tires to get flat-out and blow, that will make the damage system simulate some Visual Effects that can be configured below.
Wheel Damage Intensity
This field indicates the how much fast the wheel's tire will become flat, the higher the value the much damage a tire gets and vice-versa.
Explosion Force
The force of the Tire Explosion Field.
Rim Edge Emission Colour
The emission colour that will be visualized at the wheel's rim edges when at higher rates of heat caused by the friction between the metal of the rim and the ground the wheel is sitting on.
Rim Custom Emission Property
The emission property of the wheel's material, this property can be added when using a custom shaders to render the wheel object. The MVC systems can handle standard Unity shaders, HDRP and URP standard shaders as well. When using a custom built shader you'll have to assign this field in order to visualize the damage effect otherwise an error will be thrown in the Console window .

Camera Preset
A camera preset is an instance that contains the necessary information about the Vehicle Follower's behaviour. You can switch between presets at runtime using the Camera Switch Input. The preset settings can differ from a preset to another depending on their enabled features.
Name
The camera name is the preset identifier and cannot be a duplicate of another preset.
Type
The type field indicates the main behaviour type of the preset. A camera could be a Follower or a Pivot.
Features
Each camera preset should has its own set of features depending on its Type. Here's a full list of features and which types of cameras are compatible with:
  • Vertical Orbit: The camera can rotate around the vehicle's X axis. Available for Follower and Pivot type of presets.
  • Horizontal Orbit: The camera can rotate around the vehicle's Y axis. Available for Follower and Pivot type of presets.
  • Speed Pulse: This features allows the camera to go a bit backwards at higher speeds. Available for Follower type of presets.
  • Speed FOV : This feature allows the camera to feel like going backwards at higher speeds, by just changing its Field Of View . Available for Pivot type of presets.
  • Gear Pulse: This features allows the camera to go a bit forward in a fast way while shifting gears, so the player can get the feeling of something happening. Available for Follower type of presets.
  • Sideways Slip Pulse: The camera can move left and right while the target vehicle is slipping on its X axis. Available for Follower and Pivot type of presets.
  • Steering Tilt: This feature will allow the camera to rotate around its Z axis, this rotation depends on the vehicle's velocity at the X axis. Available for Follower and Pivot type of presets.
  • NOS FOV : While the vehicle's NOS is active, the player will feel like the camera is moving backward in an intense way, by just changing its Field Of View . Available for Follower and Pivot type of presets.
  • Speed Shaking: While running at higher speeds, the camera will simulate some shaking effects with a certain intensity. Available for Follower and Pivot type of presets.
  • Off-Road Shaking: Off-Road grounds can be configured in the Grounds & Surfaces panel, and if the target vehicle is on top of one of these grounds and this feature is enabled, the player will get that feeling of the vehicle running on top of something rough. Available for Follower and Pivot type of presets.
  • NOS Shaking: The camera will shake randomly in case the vehicle's NOS is active. Available for Follower and Pivot type of presets.
  • Sideways Slip Shaking: As the vehicle's absolute sideways slip value gets higher, the camera starts to shake randomly. Available for Follower type of presets.
  • Forward Slip Shaking: The camera will shake in case the vehicle's forward slip value is positive. The higher the slip value, the intense the shaking is. Available for Follower type of presets.
  • Brake Slip Shaking: The camera will shake in case vehicle's forward slip value is negative. The lower the slip value, the intense the shaking is. Available for Follower type of presets.
  • Showcase: When the target vehicle is idle for a period of time, the camera will rotate around the vehicle in slow-motion to visualise it in a cinematic way. Available for Follower type of presets.
  • Stunts Showcase: While doing stunts with the target vehicle like donuts , the camera will stay in an idle position for a better cinematic view. Available for Follower type of presets.
  • Obstacles Detection: This feature will make the camera dodge intensely in case its target vehicle collides with an object or another vehicle. Available for Follower and Pivot type of presets.
Pivot Position
This field indicates whether a camera pivot should be situated Inside a vehicle like an FPS camera, or Outside of it, like a Bumper or a Hood camera. This configuration field is only available for Pivot type of cameras.
Pivot
For camera presets type of Follower, the pivot indicates the position of on which the Vehicle Follower should focus on and rotate around, the pivot position is always relative to the vehicle position and rotation, so you can choose separately on each axis the option you want as the following:
  • X: Left, Centre and Right
  • Y: Top, Middle and Bottom
  • Z: Front, Middle and Rear
Impact Shaking

Engine Folder Setup
Every Engine Preset can have a folder where it can store audio clips of the engine sound and later used by the MVC audio systems. These folder can be located under the main Audio Folder, within the Engines Folder.
The default path of the mentioned directory is the following: Assets/Resources/<Audio Folder>/<Engines Folder>/<Engine Name>.
Example: Assets/Resources/Audio/Engines/V8NM 4.0L P60B40
The structure of the engine folder is the following:
  • <Engine Name>: The main engine directory

    • Engine: This folder contains the vehicle's engine sound. (Optional in case the Exhaust folder is available)

      • <Engine Audio Clips Here>: Acceleration/Deceleration audio clips. (Optional in case of using Accel & Decel folders)

      • Accel: For acceleration audio clips. (Optional in case of using audio clips inside the Engine folder directly)
        • <Engine Acceleration Audio Clips Here>
      • Decel: For acceleration audio clips. (Optional - See Accel folder notice above)
        • <Engine Deceleration Audio Clips Here>

    • Exhaust: This folder contains the vehicle's exhaust sound. (Optional in case the Engine folder is available)

      • <Exhaust Audio Clips Here>: Acceleration or Deceleration audio clips. (Optional in case of using Accel & Decel folders)

      • Accel: For acceleration audio clips. (Optional in case of using audio clips inside the Exhaust folder directly)
        • <Exhaust Acceleration Audio Clips Here>
      • Decel: For acceleration audio clips. (Optional - See Accel folder notice above)
        • <Exhaust Deceleration Audio Clips Here>
It's unnecessary to have an Exhaust folder for an Electric engine as exhaust sounds are not available for EVs in real-life.
Audio clips inside the specified folders should meet the following naming requirements in order to get recognized by the MVC audio systems:
Every audio clip represents a RPM level of the sound.
To separate between numbers and unrelated text within the audio clip name, the audio systems uses a Name Separator, which is defined in the Sound Effects panel. The default value of this separator is _.
The separator can be located anywhere in the audio clip name, just make sure it separates the RPM value.
You can also use only RPM values to name your audio clips.
Here's a quick recap of the requirements mentioned above:
  • engine-name_<RPM>.wav
  • engine_name_<RPM>.wav
  • <RPM>_engine_name.mp3
  • <RPM>_engine-name.wav
  • some_text_<RPM>_engine_name.mp3
  • <RPM>.wav
Here are some wrong examples that are incorrect:
  • engine-name<RPM>.wav
  • <RPM>engine_name.wav
  • <RPM>-engine_name.wav
  • some_text-<RPM>_engine_name.mp3
  • abcd<RPM>.wav

Supercharger Preset
Superchargers are a famous type of engine aspiration components that blows compressed dense air into an internal combustion engine to get more power.
Supercharger Type
Superchargers have a variety of types:
  • Roots: One of the Positive Displacement chargers. This type of superchargers is commonly used in old American Muscle cars with Hood-blowers . In real-life, this type gives lower-boost values compared to Twin Screw superchargers due to its old-tech components.
  • Twin Screw: One of the Positive Displacement chargers. This type of superchargers is usually used by the majority of vehicles. In real-life, this type gives higher-boost values compared to Roots superchargers due to the usage of high-tech components.
  • Centrifugal: A Centrifugal supercharger is very similar to a turbo in terms of performance, except it uses a belt linked to the engine crank instead of using exhaust gases to drive the charger. This type of superchargers provide high levels of boost at high RPMs (higher than Positive Displacement superchargers in real-life) and lower boost levels at low RPMs, just like Turbochargers.
Minimum Boost
The supercharger boost multiplier at low RPMs. (Only available for Centrifugal superchargers)
Maximum Boost
The supercharger boost multiplier at high RPMs.

  • Quad: Quad means four turbos.

Minimum Boost
The turbocharger boost multiplier at low RPMs.
Maximum Boost
The turbocharger boost multiplier at high RPMs.

Particle Systems
Particle effects are an essential part of the Visual Effects system. The following parameters control particles behaviour & shape at Runtime.
Leaving any field as empty will lead the VFX system to ignore it and should not cause any errors.
Exhaust Smoke
For vehicles with Diesel engines, this particle system will be emitted depending on how much a vehicle Fuel Pedal is engaged.
Exhaust Flame
Unlike Diesel engines, Petrol or Gas engines throw flames instead of smoke. This particle effect can be emitted in case a vehicle's Exhaust Effects flag is enabled. The particle system is emitted randomly depending on the vehicle Emission probability value, when the Fuel Pedal is released, a gear shift has occurred or when the engine RPM exceeds the Over-Rev RPM.
NOS Flame Type
To prevent resource redundancy, we though maybe the NOS Flame could override the original Exhaust Flame using two methods:
  • Particle System: By choosing this option, the Exhaust Flame will be overridden by another Particle System prefab.
  • Material: If you prefer to just override the Exhaust Flame particle material, this option will do the job for you.
You'll have a field to assign below this parameter depending on your type choice.
NOS Flame Particle System
If NOS Flame Type is set to Particle System, a particle system prefab is required to override the original Exhaust Flame.
NOS Flame Material
In case NOS Flame Type is set to Material, a material asset would be enough to save resources and generate a new particle system based on the original Exhaust Flame.
Damage Sparks
When a vehicle chassis is making constant contact with another object, a sparks particle effect will be emitted at the contact point.
To use this feature, the Damage system has to be enabled.
Tire Explosion
At the moment where a vehicle tire becomes fully damaged, the mentioned particle system will be emitted.
For this feature to work, both Damage & Wheel Health systems have to be enabled.
Tire Explosion Force Field
Once the Tire Explosion particle effect is emitted, a Particle System Force Field could be created to visualize the intensity of the event by pushing the Ground Particle Effect away from the centre of the explosion (generally, tires burnout smoke).
The Tire Explosion field has to be assigned for this feature to work properly.
AI Path Visualizer
While AI Vehicles could predict where to use brakes on its own, this job might be difficult for beginner players. For this, the MVC has its own AI Path Visualizer implemented, to tell the player whether to brake or accelerate to keep his own vehicle on track.
Renderer Material
The Path Visualizer renderer material.
Ground Offset
The Visualizer offset from its target surface.
Width
The Path Visualizer mesh width.
Heat Gradient
This colour gradient defines how the AI Path Visualizer should appear. The colour to the extreme left is for acceleration, while the colour to the extreme right is for braking.
Wheel Marks
Wheels marks basically are tire trails on a specific type of surface.
Resolution
The wheel mark level of detail.
Increasing Resolution to higher values, might cause some performance drops without making any noticeable visual changes.
Ground Offset
The wheel contact point offset. This method is used to prevent wheel marks from interfering with ground surfaces, therefore preventing visual problems. Measured in Metres m or Feet ft.
Occlusion Distance
The distance between the mark mesh & the main camera at which a wheel mark renderer gets disabled to gain performance.
Lights
This settings section controls vehicles light behaviour.
Leaving any field as empty will lead the VFX system to ignore it and should not cause any errors.
Indicators At Engine Stall
This indicates whether indicator lights should be enabled when a vehicle engine stalls.
Indicators At Fog Time
Should vehicle indicator lights be enabled when the Fog Time toggle is set to On.
Exhaust Flame Light
A light prefab to be created along with a Exhaust Flame or a NOS Flame to simulate fire lighting.
Side Signals Period
The time period it take for side signal lights (indicators) to complete an On & Off cycle.
Lamp Lights Damping
The speed at which lights of type Lamp switch between On & Off states visually.
Custom Emission Property
For lighting mesh renderers that has materials using a custom made shader, this key field refers to the emission property of the mentioned shader.
Wheel Spin Effect
A set of configurations for the Vehicle Wheel Spin Effect component.
RPM Threshold
The wheel speed threshold at which the spin effect is fully spread across a full circle (a full the spread angle is equal to 360 degrees).
Shutter Amount
The radial blur intensity multiplier.
Mesh Instancing
Use this feature to improve runtime performance using GPU Instancing .
Samples Count
The number of mesh clones the toolkit should draw at runtime.
Alpha Offset
The transparency/opacity offset of the spin mesh material.
Brakes Heat
For brakes friction simulation, the Multiversal Vehicle Controller provides the Brakes Heat system.
Emission Color Property
The material emission property key for brake disks mesh renderers.
Heat Emission Color
Colour of brake disks at a maximum heat level.
Heating Speed
The speed intensity at which the brake heat increases.
Cooling Speed
The speed intensity at which the brake heat decreases.
Clamp Brake Heat
Enabling this flag will max out the brake heat level and prevent it from exceeding the maximum level.
Affect Performance
Setting this flag to On will make the physics system consider the heat factor as a performance downgrade for brakes.
Editor Settings
The last things left to configure are Editor settings. Using the parameters mentioned below, you can modify how the toolkit visualize its different features inside the Unity Scene View and Inspector , as well as the toolkit's Settings Panel.
Auto Update Check
Enables and disables the toolkit automatic update detection. If this parameter is disabled, you can check for updates manually through the Check For Updates button below.
Values Unit
Same as Runtime Units, except it's for editor windows & inspectors.
Power Unit
The editor unit for Power values.
Torque Unit
The editor unit for Torque values.
Spring Target Measurement
Vehicle suspension Spring Target measurement type. This parameter could be of two values:
  • Target: Or better say the target percentage.
  • Length: Will make the Spring Target measurable in Millimetres mm. This value will be clamped to the vehicle suspension Spring Length.
Exhaust Simulation FPS
When simulating exhaust flames in edit mode, you would not want the editor FPS to drop below a certain value. This parameter defines the simulation FPS limit.
Preview Suspension Adjustment
Enabling this feature will let you see changes to your vehicles suspension in edit mode within the Scene View .
Gizmos
While working with MVC inside Unity, our systems will try to visualize some important details inside the Scene View . The following parameters define the look of these visual elements inside Unity:
  • Gizmos Size: The size multiplier of the toolkit gizmos.
  • COM: A coloured sphere indicating the position of a vehicle centre of mass.
  • Engine: A coloured sphere indicating the position of the engine.
  • Exhausts: A coloured sphere indicating the position of the vehicle exhaust output.
  • Wheel Marks: Coloured lines visualizing Wheel Marks paths.
  • Headlights: Some coloured spheres indicating the position of a vehicle headlight sources.
  • Side Signal Lights: Same as Headlights, except for side signal & indicator lights.
  • Interior Lights: Coloured spheres indicating the vehicle interior lights position. Same as Headlights & Side Signal Lights.
  • Rear Lights: Coloured spheres indicating the vehicle rear lights position. The same as Headlights, Side Signal Lights & Interior Lights.
  • IK Pivots: Some coloured spheres referring to Vehicle IK Pivots' position.
  • Audio Reverb Zone: A box/sphere indicating an Audio Zone of type Reverb.
  • Darkness Weather Zone: A box/sphere indicating an Weather Zone of type Darkness.
  • Fog Weather Zone: A box/sphere indicating an Weather Zone of type Fog.
  • Wheels Damage Zone: A box/sphere indicating an Damage Zone of type Wheels.
  • Damage Repair Zone: A box/sphere indicating an Damage Zone of type Repair.
  • AI Path Bezier: The AI Path gizmos colour.
  • AI Path Bezier Selected: The AI Path gizmos colour at its own state of selection.
  • AI Path Anchor: A coloured sphere indicating the positions of the AI Path points.
  • AI Path Start Anchor: A coloured sphere that refers to the AI Path start point.
  • AI Path Moved Anchor: A coloured sphere that refers to the AI Path moved points due to interference with obstacles.
  • AI Path Failed Moving Anchor: A coloured sphere that refers to the AI Path points that have failed being moved or aligned to the ground.
  • AI Path Controls: Some coloured lines & handles that you can use to modify & stance a specific AI Path.
  • AI Brake Zone: A box/sphere indicating an AI Zone of type Brake.
  • AI Handbrake Zone: A box/sphere indicating an AI Zone of type Handbrake.
  • AI NOS Zone: A box/sphere indicating an AI Zone of type NOS.
  • AI Obstacle Bounds: A coloured box indicating AI obstacles bounds.
  • AI Obstacle Sensors: Some coloured lines referring to AI Vehicle's obstacle sensors.
  • AI Obstacle Sensors Active: The AI Vehicle sensors colour when hitting an obstacle.
